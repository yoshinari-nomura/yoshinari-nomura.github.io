ごめんなさい。ちょっと時間が取れたもので、嬉しくなったのと、
土屋さんに刺激されたので、昨夜遅くまで書いてました。

mhc-fake.el ですが、例えばですね、
mnews 用の実装や、生の mh に皮を被せて使いたい人もいるかもしれま
せん。そういう人が、「ちょこっとハックしてみたらできちゃった」と
いうようなとっつきやすさも重要だと思っているんですよ。もともと 
mhc 自体がそうやって生まれていますし、Gnus や Wanderlust に対応
しようと思われた方がいらっしゃったのは、このお手軽 *そう* に見え
る所があったからだと思っています。つまり、Mew に特化して、どうに
かこうにか実現している訳ではないことを直感させるような顔付きをし
ていたためでしょう。その顔が有名な MH だったら、多少不格好でも親
しみが湧きますよね。

僕らの知らない所で、mailagent や procmail を使って、面白い事をし
ようと思ってる人がいるかもしれませんし、IMAP サーバに管理させよ
うと思っている人がいるかもしれません。MH 形式というのは、歴史が
あるだけに、思わぬ応用が考えられたりするんではないかと思っていま
す。その可能性を残しておきたいなぁという気分です。

それと、sync のためだけにしては、全体としての作業量が多いような
気がしませんか? それによって、何か大きい物が *ユーザ* の手に入る
なら別ですが。

mhc-sync が 10行足らずのコードを追加するだけで、今回の問題を妥当
な効率で解決していることを考えると、sync の方で頑張るのがいいの
ではないでしょうか。

他の皆さんの意見はいかがでしょう。

実は、この案と似た案は、僕自身が 1回 Mew に入れようとして却下さ
れたんですよ。なので、プログラマから見た利点は承知しているつもりです。

例えば、どうせ変更するなら、この際、データベースを外に追い出して、
小さな NTTP (や IMAP) server を模したサーバに管理させるというの
はどうでしょう。これも親しみのある顔立ちをしてますし、Id はばっ
ちり管理してくれますし、グループウェア的な共有もできそうですし。

# そう、Mew2 は IMAP server 見立てた private な server でやろうっ
# ていう話をしたのも僕でした。
